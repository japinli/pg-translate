    索引扫描并不意味着高效

    我经常看到人们在查看执行计划时忽略的一个问题：他们的所有扫描都涉及到索引，因此他们认为查询可能已经尽可能快（或高效）了。

    大部分人都知道，如果顺序扫描中有大量的行被过滤掉，则表明索引可以提供帮助。但是索引加快速度的原因同样适用于具有过滤更多行的其他扫描，其中过滤行更多的索引可以加快速度！虽然我们经常会看到索引过滤器显示 Rows Removed by Filter，但遗憾的是，有些情况 EXPLAIN 并不会报告（我们稍后会讲到）。

    在本文中，我们将通过几个示例以及一些需要注意的事项来说明如何加快速度。

    值得注意的是，还有其他方法可以进一步优化索引的使用。我们还通过 pgMustard 中的 Index Efficiency 提示重点介绍了一个这样的例子，即通过索引重新检查删除了大量行。但这篇文章已经够长了，所以我们改天再讨论。

    解决 PostgreSQL 中的重叠查询问题
